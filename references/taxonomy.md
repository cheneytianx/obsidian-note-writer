# Vault 目录分类体系（Taxonomy）

## 推荐目录结构

```
vault/
├── inbox/             # 临时收集，待整理的原始笔记
├── topics/            # 长期关注的主题/领域
│   ├── tech/          # 示例：技术
│   ├── reading/       # 示例：读书
│   └── work/          # 示例：工作
├── resources/         # 工具、模板、术语、参考资料
├── daily/             # 日记 / 日志（格式：YYYY-MM-DD.md）
└── archive/           # 已完成、过期或不再维护的内容
```

> 这是面向**轻量使用场景**的推荐结构，不是强制。若用户 vault 已有自己的结构，以用户结构为准。
>
> 对于轻量使用场景，不建议一开始就细分过多目录。
> 文件夹只负责粗分类，具体主题关系优先通过标签、双链和主题索引页维护。
> 当某个主题下的笔记数量明显增多时，再考虑拆分子目录。

## 适用场景

- 只关注少数几个主题/领域进行持续记录
- 不想花太多精力思考“这条笔记该放哪个文件夹”
- 更依赖 Obsidian 的标签 / 双链 / 搜索，而不是深层目录

如果使用场景更复杂（多项目并行、有明确截止日期的任务管理、大量参考资料需要精细分类），再考虑升级为 PARA 等更完整的结构。

## 分类决策树

```
这是什么类型的内容？
├── 今天发生的 / 日志        → daily/YYYY-MM-DD.md
├── 工具、模板、术语、速查表 → resources/xxx.md
├── 长期关注主题下的笔记     → topics/<主题>/话题.md
│   （如 topics/tech/、topics/reading/、topics/work/）
├── 已完成 / 过期 / 不再维护 → archive/...
└── 不确定                   → inbox/话题.md（后续再分类）
```

## 主题目录的使用建议

- `topics/` 下的子目录 = 你**长期关注**的主题，通常不超过 5 个
- 主题数量很少时，可以不建子目录，直接在 `topics/` 下放笔记
- 某个主题笔记变多时，再考虑：
  - 拆分子目录，例如 `topics/tech/frontend/`
  - 或建一个主题索引页 `topics/tech.md`，用双链组织内容

## 文件命名规范

| 规则 | 示例 |
|------|------|
| 小写，空格用连字符 | `gradient-descent.md` |
| 中文内容可用中文名 | `梯度下降.md` |
| 日期类固定格式 | `2026-05-09.md` |
| 论文/书用原标题 | `attention-is-all-you-need.md` |
| 避免序号前缀 | ~~`01-intro.md`~~ |
| 避免过于宽泛 | ~~`notes.md`~~ ~~`misc.md`~~ |

## 多话题笔记的处理

一次对话涉及多个独立话题时，优先**拆分**成多个笔记并互相双链，而不是合并成一个大文件。

例外：话题紧密相关、总长度 < 300 行时可以合并。

## 目录设计原则

1. **一级目录控制在 5 个左右**，避免选择成本过高
2. **先用少量大类，后续再拆**，不要一开始就预设很多分类
3. **文件夹负责“位置”，双链和标签负责“关系”**
4. **主题数量少时，不必每个主题都建文件夹**
5. **优先维护主题索引页（MOC）**，比多层文件夹更灵活
